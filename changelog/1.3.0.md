<a name="Overview"></a>

## Overview

Revision 1.3.0 is a maintenance revision that removes several deprecated types and stabilizes the 1.x feature set in preparation for Value.* updates planned later in the year. This revision also finalizes preparation for the TypeScript 7 native compiler release, in which all TypeBox inference paths are trimmed down to their core essentials to ensure the best possible inference performance for `Type.Script`.

Despite these deprecations, this revision is expected to be very low impact. A major version bump has been deferred until the aforementioned Value.* work has been finalized. The planned Value.* updates are expected to include breaking API changes to Encode and Decode along with some potential changes to StaticEncode, StaticDecode. 

TypeBox will defer 2.x until Value.* functions are finalized.

<a name="Base"></a>

## Base

The Base type was originally added in the 1.x release to be used as a hook to try and support Standard Schema. However early into the 1.x release, it became clear that supporting class instances embedded in JSON Schema was going to be untenable in the longer term, both from a type-mapping standpoint and from a runtime type computation standpoint, particularly when handling structured clones of user-defined JavaScript Base class instances.

TypeBox has discontinued support for Standard Schema since 1.0.28, both because of the complexity cost of hosting the specification within JSON Schema was too high, and because the specification conflicts with the standardization efforts TypeBox follows, specifically JSON Schema under IETF and ECMA.

TypeBox will reject class instances embedded in schematics moving forward as a core design constraint, however an external reference adapter to Standard Schema can be found at the link below.

[Reference: Standard Schema](https://github.com/sinclairzx81/typebox/tree/main/example/standard)

### Use Unsafe / Refine Instead of Base

As the Base type was also servicing as an extension mechanism for user defined types in early 1.0.x, implementations should update to use Unsafe / Refine moving forward. The previous Base fallbacks for Uint8Array and Date have been updated to use the Unsafe / Refine pairing at the links below.

- [Reference: Uint8Array](https://github.com/sinclairzx81/typebox/blob/main/example/legacy/uint8array.ts)
- [Reference: Date](https://github.com/sinclairzx81/typebox/blob/main/example/legacy/date.ts)

A reference implementation for Uint8Array is as follows:

```typescript
import Type from 'typebox'

export type TUint8Array = Type.TUnsafe<globalThis.Uint8Array>

export function Uint8Array(): TUint8Array {
  return Type.Refine(Type.Unsafe<globalThis.Uint8Array>({ type: 'Uint8Array' }), 
    value => value instanceof globalThis.Uint8Array,
    _value => 'Expected Uint8Array')
}
```

<a name="Awaited-Promise-AsyncIterator-and-Iterator"></a>

## Awaited, Promise, AsyncIterator, and Iterator

Revision 1.3.x removes these types in favor of structural composition via TypeBox's existing type system. Moving forward, TypeBox will consider all JavaScript built-ins to be higher-order constructs that can be composed from simpler types. TypeScript models JavaScript built-ins in the exact same way.

These specific types date back to earlier versions of 0.x, where they were originally added to model RPC function return types. The intention was to describe callable RPC interfaces; however, a consumer of an RPC service rarely needs to know if a RPC server function is synchronous or asynchronous (it's always asynchronous from the callers perspective), and where other aspects such as iteration and streaming is best described against the capabilities of the RPC protocol, not TypeBox's definition of a JavaScript iterator.

To highlight composition, the following example models `lib.d.ts`-style definitions for `Promise` and `Date` using TypeBox's existing primitives, then uses the resulting `Promise` type to describe an RPC-style function. The aspect to consider here is that `Promise<T>` can be modelled through structural composition, which is much better than TypeBox carrying around intrinsic definitions for these types.

```typescript
// -----------------------------------------------------------
// Model various lib.d.ts definitions
// -----------------------------------------------------------
const lib_d_ts = Type.Script(`
  type Promise<T = unknown> = {
    then: (onFulfilled: (value: T) => unknown) => unknown;
  }

  type Date = {
    getTime: () => number
    getDate: () => number
    getTime: () => number
    getFullYear: () => number // etc, etc
  }
`)

// -----------------------------------------------------------
// Application Types
// -----------------------------------------------------------
const { Add } = Type.Script(lib_d_ts, `

  type Vector = { x: number, y: number }

  type Add = (a: Vector, b: Vector) => Promise<Vector> // from lib.d.ts
`)

// Example
declare const add: Type.Static<typeof Add>

const { x, y } = await add(      // { x: number, y: number }
  { x: 1, y: 0 }, 
  { x: 0, y: 1 }
) 
```

The `Function` and `Constructor` types will remain as part of TypeBox's Type IR extensions to JSON Schema.

<a name="Mutate"></a>

## Mutate

The Mutate type was introduced in 0.27.x to handle updates to React state without triggering a redraw. The function was written for that specific use case, and its scope is generally too narrow for inclusion in TypeBox; a utility like this is better suited to React itself.

A fallback implementation is being retained under `example/mutate/*`, which can be copied into projects that still require it; however, its usage across the ecosystem is believed to be minimal to nonexistent.
